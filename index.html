Campus Parking Platform

Full Stack Web Dev Project • Jan – Jun 2025

Why we built it

Finding campus parking is frustrating: scattered rules, unclear permit limits, and no single place to see what’s allowed. Our platform makes it simple, one app to buy permits, pay tickets, and see your status in real time. For admins, it’s the control center they’ve always wanted: set limits, override prices, and manage accounts with confidence.

Tech Overview

  • Frontend: Next.js + TypeScript
  • APIs: GraphQL + REST (type‑graphql, Express/tsoa), JWT + Google OAuth
  • Data: PostgreSQL per domain with JSONB for flexible models
  • Payments: Stripe Checkout + webhooks
  • Infra: Docker containers behind NGINX (SSL) on AWS EC2
  • Quality: TDD with Vitest, Supertest, and React Testing Library

How it works (short & clear)

  • Architecture: microservices for Auth (REST), Permits/Vehicles/Tickets (GraphQL), and Payments/Stripe (REST), NGINX reverse‑proxy in front.
  • Auth: email/password uses JWT (http-only cookie) for session, Google OAuth provides SSO. Protected routes verify tokens server-side.
  • Permits/Tickets: Admin sets zone limits & pricing. Driver requests a permit → server validates availability & price → creates a pending order.
  • Payments: server creates a Stripe Checkout Session, user pays on Stripe, webhook confirms payment and activates the permit.
  • Admin Control: Admin UI lists all accounts, can suspend/enable users and override pricing/limits. Changes persist in PostgreSQL and show up via service APIs.
  • Testing/Quality: TDD with Vitest/Supertest/RTL, type-safe endpoints with TypeScript.
  • Deploy: Dockerized services on AWS EC2, NGINX handles SSL/HTTP2 and routes to containers.

Architecture

Key Flows

  • Buy Permit: Driver selects vehicle/type → Permit validates via Vehicle, checks zone capacity, creates order, Stripe payment completes and webhook activates permit, Mailgun sends receipt.
  • Issue Ticket: Enforcement checks plate via Permit, Ticket service creates ticket and emails driver, Admin can override or resolve challenges.
  • Admin Controls: Manage zone capacities, prices, user status, role‑gated dashboards with real‑time lists.

Design Choices

  • JSONB in Postgres: faster schema evolution, traded strict constraints for iteration speed.
  • Direct HTTP calls: simple discovery in a monorepo, future gateway/registry for scale.
  • Role‑gated GraphQL: @Authorized and resolver checks for clear boundaries.

CineSync Movie App

Software Engineering Project • Jan – Jun 2025

Mobile app for group movie nights. Built a real-time, multi-user workflow with auth, groups, invites, and shared watchlists. Emphasis on state consistency and UX polish.

React NativeExpo Firebase (Auth/Store)Realtime Design SystemsAgile
Core Features & How It Works

Sign‑in & Sessions

  • What it does: Email/password and “Continue with Google.” Sessions stay signed‑in across app restarts.
  • How it works: Firebase Auth issues an ID token, we refresh it automatically and read simple role claims (owner/mod/member).
  • Why this matters: Low friction for users, and predictable permissions for the app.

Groups & Access

  • What it does: Users create or join groups to plan movie nights together.
  • How it works: Each group has members with roles. Firebase Security Rules allow only members to change group data, owners can toggle fairness or manage members.
  • Why this matters: Permissions are enforced server‑side, so UI mistakes don’t become security issues.

Invites & Onboarding

  • What it does: Invite friends by email, they can accept or decline. Pending invites show up as a small badge.
  • How it works: An invite document is created, when it’s accepted we add the member to the group and bump a lightweight counter that drives the badge.
  • Why this matters: Onboarding is reliable and the badge updates instantly without heavy reads.

Watchlist, Voting & Fairness

  • What it does: Add movies to a shared watchlist, like them, mark them as watched, and optionally enable a fairness toggle so picks don’t get dominated by one person.
  • How it works: Actions update the UI immediately (optimistic UI). If the write fails, the last confirmed state is restored from the live listener.
  • Why this matters: The app feels instant but stays correct even on flaky networks.

Search & Add to Group

  • What it does: Search a movie catalog and add a title to your group’s list.
  • How it works: When you pick a result, we save one canonical movie record (title, year, IDs) and then link it into your group’s watchlist. The same film can appear in many groups without creating duplicates.
  • Why this matters: Clean data (no duplicates) and faster queries later (one source of truth).

Sorting & Filtering

  • What it does: Order by likes or Rotten Tomatoes score, filter by genre.
  • How it works: We keep a small client‑side index to combine signals without expensive multi‑field queries. Results are memoized so scrolling stays smooth.

Realtime Experience

  • What it does: Everyone sees changes as they happen, new movies, likes, invite counts.
  • How it works: Live listeners stream updates to the screen. We preserve scroll position and resync after reconnecting.

Reliability & Shipping

  • What it does: Keeps critical flows stable as features ship.
  • How it works: Component/integration tests cover invites, watchlist actions, and fairness. Releases go out with Expo EAS channels so we can stage changes.

Example: Add a Movie → Visible to the Group

  1. Search returns results from the external API.
  2. Select saves one canonical movie record and links it to your group’s watchlist, so the same title isn’t duplicated across groups.
  3. Instant feedback: the movie appears immediately, if the server rejects the change we revert to the last confirmed state.
  4. Realtime: everyone in the group sees the update without refreshing, and badges update via a lightweight counter.

Generative AI – Six Thinking Hats & Clustering

Full‑Stack AI/NLP Project

I built this to explore bias in AI answers and ways to make responses more balanced. The app gathers six short viewpoints (the “Six Thinking Hats”) for any prompt, adds a critical check for balance, and then groups the text to see patterns in tone and themes. The goal was simple, clearer answers with fewer one‑sided takes.

PythonFlaskOpenAI PromptingClusteringSentiment
Big Picture & Findings

Purpose

  • Explore how prompts shape bias in AI answers
  • Find simple ways to get more balanced responses

How it works

  • Generate six short perspectives (hats) for each prompt
  • Add an extra critical take to check optimism and balance
  • Group similar ideas and compare tone to spot skew

Experiments

  • With vs. without roles to see variety and coverage
  • Measure whether an added critique shifts overall balance

What I learned

  • Structured prompts surface more angles and reduce one‑sidedness
  • A small critique step further balances positive/negative tone
  • Lightweight grouping is enough to visualize bias patterns
